GDB (GNU Debugger) Overview

GDB is a powerful debugging tool for programs written in languages like C and C++. It allows you to inspect and control the execution of a program to find and fix bugs. You can use GDB to:

  • Run a program with controlled execution.
  • Set breakpoints to pause execution at specific points.
  • Inspect and modify memory, variables, and registers.
  • Step through code line by line.

Using set variable in GDB

The set variable command allows you to manually modify the value of a variable while debugging. The syntax is:

gdb
set variable <variable_name> = <new_value>

For example, if you have a variable x in your program, you can modify its value in GDB:

gdb
(gdb) print x
$1 = 10
(gdb) set variable x = 20
(gdb) print x
$2 = 20

Key Aspects of set variable

  1. Scope: It only affects the variable in the current stack frame or context.
  2. Effect Duration:
    • If the program continues executing, the modified value will persist until it is reassigned by the program.
    • If the variable goes out of scope (e.g., if its a local variable and the function returns), it will no longer be accessible.
    • If the variable is global or static, the change will remain until modified again or the program exits.
  3. Usage in Conditional Debugging:
    • You can use set variable to alter execution behavior without modifying the source code.

How Long is the Set Value Used?

  • If the variable is global or static, the new value persists for the entire program execution unless it is changed by the program itself.
  • If the variable is local, the change persists until the function returns or the stack frame is destroyed.
  • If the program restarts, the change is lost because GDB does not retain modifications between runs unless explicitly saved.

Step-by-Step Guide to Debugging a C Program with GDB

    return 0;

The program tries to dereference a NULL pointer (p), causing a segmentation fault.

Before debugging with GDB, compile the program with the -g flag:

  • -g: Includes debug symbols in the binary (needed for GDB to inspect variables and source code).

bash

public:
    int data;

    Node(int val) : data(val), next(nullptr) {}
};

class LinkedList {
public:
    Node* head;

    LinkedList() : head(nullptr) {}

    void addNode(int val) {
        Node* newNode = new Node(val);
        newNode->next = head;
        head = newNode;
    }

    void printList() {
        Node* temp = head;
        while (temp != nullptr) {
            std::cout << temp->data << " -> ";
            temp = temp->next;
        }
        std::cout << "NULL" << std::endl;
    }

        Node* temp = head;
        while (temp != nullptr) {

            delete temp;
            temp = nextNode;
        }

    }

Illustrating Makefile Commands with a Step-by-Step Example

We will create a Makefile for the following project structure:

pgsql
/home/labDirectory/
 file1.cpp  (Contains `main()`)
 file2.cpp  (Contains `main()`)
 lib1.cpp   (No `main()`, compiled into object file)
 lib2.cpp   (No `main()`, compiled into object file)
 lib3.cpp   (No `main()`, compiled into object file)
 lib1.h
 lib2.h
 lib3.h
 Makefile

1 Understanding the Requirements

  • We compile the 3 non-main() C++ files into object (.o) files.
  • We link these object files with file1.cpp and file2.cpp to create two executables.
  • The compilation uses:
    • --std=c++11 when running make normally.
    • --std=c++11 -Wall -Werror if additional flags are given (make CXXFLAGS="...").
  • The final directory must contain only:
    • Original 9 source files
    • 3 .o files
    • 2 executables

2 Writing the Makefile

make
# Compiler settings
CXX := g++
CXXFLAGS := --std=c++11  # Default flag (can be overridden)

# Source files
SRCS := file1.cpp file2.cpp lib1.cpp lib2.cpp lib3.cpp
HEADERS := lib1.h lib2.h lib3.h

# Object files (only for non-main files)
OBJS := lib1.o lib2.o lib3.o

# Executables
EXE1 := file1.out
EXE2 := file2.out

# Default target (Runs when `make` is executed)
all: $(EXE1) $(EXE2)

# Rule for creating executables (each depends on object files)
$(EXE1): file1.cpp $(OBJS)
	$(CXX) $(CXXFLAGS) -o $@ $^

$(EXE2): file2.cpp $(OBJS)
	$(CXX) $(CXXFLAGS) -o $@ $^

# Rule for compiling object files
%.o: %.cpp $(HEADERS)
	$(CXX) $(CXXFLAGS) -c $< -o $@

# Clean target (removes generated files)
clean:
	rm -f $(OBJS) $(EXE1) $(EXE2)

3 Explanation of Each Rule

all: $(EXE1) $(EXE2)

  • Default target. When you run make, it compiles both file1.out and file2.out.

$(EXE1): file1.cpp $(OBJS)

  • Creates file1.out
  • Uses file1.cpp and the three object files.
  • $@ The target (i.e., file1.out).
  • $^ All dependencies (file1.cpp lib1.o lib2.o lib3.o).

$(EXE2): file2.cpp $(OBJS)

  • Same as above but for file2.out.

%.o: %.cpp $(HEADERS)

  • This rule compiles each .cpp file into a .o file.
  • % is a pattern-matching wildcard:
    • If lib1.cpp needs to be compiled, this rule will create lib1.o.

clean Target

  • Deletes object files and executables to reset the build:
    bash
    make clean
